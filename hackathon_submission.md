# الحارس الدلالي: مساعد ذكي للتحقق اللحظي من منطقية بيانات الاستبيانات

---

## 1. ملخص الفكرة

الحارس الدلالي نظام ذكي يعمل كمصفاة منطقية لبيانات الاستبيانات والنماذج الإلكترونية. يحلّل العلاقات بين حقول الاستمارة لحظة إدخالها — ليكتشف التناقضات التي لا تراها أدوات التحقق التقليدية: هل يُعقل أن يكون عمر الشخص 17 سنة وخبرته 8 سنوات؟ هل يتوافق مسمى "مدير موارد بشرية" مع مؤهل "ثانوي"؟

يجمع النظام بين قوة النماذج اللغوية الكبيرة (LLMs) في فهم السياق، ومحرك قواعد منطقية سريع كخط دفاع احتياطي — ليمنح كل سجل درجة ثقة رقمية وتقريرًا مفصلًا بالمشاكل المكتشفة وشدتها.

---

## 2. المشكلة

تعتمد الجهات الإحصائية والحكومية على استبيانات ميدانية لجمع البيانات، لكن أدوات التحقق الحالية تقتصر على فحوصات سطحية: هل الحقل معبّأ؟ هل القيمة رقمية؟ هل البريد الإلكتروني صحيح الصيغة؟

هذا يعني أن **التناقضات المنطقية بين الحقول** تمرّ دون اكتشاف:

- موظف عمره 17 سنة بخبرة 8 سنوات ودخل 30,000 ريال — لا يوجد حقل "خاطئ" بمفرده، لكن مجتمعةً لا تصح.
- سائق توصيل بمؤهل ابتدائي ودخل شهري 25,000 ريال.
- دوام جزئي بـ 50 ساعة عمل أسبوعيًا.

**النتيجة:**
- عمليات تنظيف يدوية مكلفة بعد جمع البيانات
- إعادة زيارات ميدانية تستهلك وقتًا وموارد
- تأخّر إصدار المؤشرات الإحصائية
- قرارات مبنية على بيانات غير دقيقة

---

## 3. الحل — كيف يعمل الحارس الدلالي؟

### المبدأ
بدلًا من فحص كل حقل على حدة، يفحص الحارس الدلالي **العلاقات المنطقية بين الحقول مجتمعة** — تمامًا كما يفعل المدقق البشري الخبير، لكن بشكل لحظي وآلي.

### خط سير البيانات (Pipeline)

```
المستجيب يعبّئ الاستمارة
        ↓
ترسل الحقول كاملة (JSON) إلى الحارس الدلالي عبر API
        ↓
   ┌─────────────────────────────────────┐
   │  طبقة التحقق الذكي                  │
   │                                     │
   │  1. محرك القواعد المنطقية (فوري)     │
   │  2. النموذج اللغوي LLM (تحليل عميق) │
   └─────────────────────────────────────┘
        ↓
النتيجة: درجة ثقة (0-1) + قائمة مشاكل مصنفة بالشدة
        ↓
تنبيه لحظي للباحث الميداني / تقرير للمحلل
```

### مثال عملي

**المدخل:**
| العمر | المؤهل | المسمى الوظيفي | الخبرة | الدخل الشهري |
|-------|--------|----------------|--------|--------------|
| 17 | ثانوي | مدير موارد بشرية | 8 سنوات | 25,000-30,000 |

**المخرج:**
```
درجة الثقة: 0.28 (غير صالح)

المشاكل المكتشفة:
  ⛔ عالية: سنوات الخبرة (8) لا تتوافق مع العمر (17) — أقصى خبرة ممكنة = 3 سنوات
  ⛔ عالية: المسمى الوظيفي الإداري لا يتناسب مع العمر والمؤهل
  ⚠️ متوسطة: الدخل مرتفع جدًا مقارنة بالمؤهل والعمر
```

## 4. التقنيات والخوارزميات

### أ) نماذج لغوية كبيرة (LLMs) مع Few-Shot Prompting
- يُرسل السجل مع أمثلة لسجلات صحيحة وخاطئة إلى النموذج اللغوي (Gemini 2.5 Flash).
- النموذج يفهم السياق الدلالي — يدرك أن "طبيب" يحتاج مؤهلًا عاليًا، وأن "مدير" في سن 17 أمر غير معتاد.
- يُرجع تقريرًا منظمًا: المشاكل، الحقول المتأثرة، شدة كل مشكلة، ودرجة ثقة رقمية.

### ب) محرك قواعد منطقية (Rule-based Engine)
- مجموعة قيود واضحة تعمل بدون اتصال بالإنترنت كخط دفاع احتياطي:
  - سنوات الخبرة ≤ العمر − 14
  - ساعات العمل الأسبوعية ≤ 72
  - الدوام الجزئي < 40 ساعة
  - المسميات التخصصية (طبيب، مهندس، مدير) تتطلب مؤهلات مناسبة
  - الدخل المرتفع مع مسمى وظيفي بسيط = تناقض

### ج) تصنيف شدة الأخطاء (Severity Classification)
- **عالية (High):** تناقض شبه مستحيل — يجب مراجعة السجل فورًا
- **متوسطة (Medium):** تعارض محتمل واضح — يستحق الانتباه
- **منخفضة (Low):** ملاحظة بسيطة غير مؤكدة

### د) درجة الثقة (Trust Score)
- كل سجل يحصل على درجة من 0 إلى 1
- السجلات تحت عتبة 0.7 تُصنف "غير صالحة" وتحتاج مراجعة

## 5. البنية التقنية

```
┌──────────────────┐     ┌─────────────────────────────┐     ┌──────────────┐
│   منصة الاستبيان  │────→│   خدمة الحارس الدلالي (API)  │────→│  LLM سحابي   │
│  (ويب / جوال)     │     │   FastAPI + Python           │     │  Gemini API  │
└──────────────────┘     │                             │     └──────────────┘
                         │  ┌───────────────────────┐  │
                         │  │ محرك القواعد المنطقية  │  │  ← احتياطي بدون إنترنت
                         │  └───────────────────────┘  │
                         │  ┌───────────────────────┐  │
                         │  │ قاعدة بيانات SQLite    │  │  ← تخزين النتائج + التعلم
                         │  └───────────────────────┘  │
                         │  ┌───────────────────────┐  │
                         │  │ لوحة تحكم تفاعلية     │  │  ← رصد + تقارير + تصدير
                         │  └───────────────────────┘  │
                         └─────────────────────────────┘
```

**مكونات النظام:**
1. **واجهة API** مبنية بـ FastAPI — تستقبل بيانات الاستمارة وتُرجع النتائج
2. **طبقة التحقق المزدوجة** — LLM للتحليل العميق + قواعد منطقية كاحتياط فوري
3. **قاعدة بيانات** — لتخزين السجلات والأخطاء وتتبع جودة البيانات عبر الزمن
4. **لوحة تحكم تفاعلية** — تعرض مؤشرات الأداء، رسوم بيانية، وتفاصيل كل سجل

هذه البنية تجعل الحل قابلًا للربط مع أي منصة استبيانات عبر API بسيط — بدون تعديل جوهري في المنصة الأصلية.

## 6. نتائج التجربة الفعلية

تم اختبار النظام على 30 سجل حقيقي من بيانات استبيان سوق العمل:

| المؤشر | القيمة |
|--------|--------|
| إجمالي السجلات | 30 |
| سجلات صالحة | 21 |
| سجلات غير صالحة | 9 |
| متوسط درجة الثقة | 0.816 |
| أقل درجة ثقة (أسوأ سجل) | 0.28 |
| زمن معالجة 30 سجل | أقل من ثانية |

**أبرز الحالات المكتشفة:**
- سجل R016: عمر 17، خبرة 8 سنوات، مدير موارد بشرية، دخل 25,000-30,000 ← درجة ثقة 0.28
- سجل R021: تعارضات متعددة في 4 حقول ← درجة ثقة 0.40
- سجل R020: تناقض عمر/خبرة + دخل غير متوافق ← درجة ثقة 0.44

## 7. الابتكار — ما الذي يميّز الحل؟

- **تحقق دلالي وليس شكلي:** بينما تفحص الأدوات التقليدية صحة كل حقل على حدة، يفحص الحارس الدلالي المنطق بين الحقول مجتمعة — كما يفعل المدقق البشري الخبير.
- **يعمل لحظيًا:** التنبيه يصل أثناء التعبئة، لا بعد أسابيع من جمع البيانات.
- **قابل للتوصيل (Plug-and-Play):** يُدمج مع أي منصة استبيانات كخدمة جانبية عبر API — بدون إعادة بناء المنصة.
- **مزدوج الحماية:** LLM للتحليل المعقد + محرك قواعد كاحتياط — يعمل حتى بدون اتصال إنترنت.
- **لا يحتاج بناء نموذج من الصفر:** يستخدم Few-Shot Prompting مع نماذج لغوية جاهزة، مما يُسرّع التطوير ويخفض التكلفة بشكل كبير.

## 8. قابلية التوسع

بعد إثبات فاعلية النظام على استبيانات سوق العمل، يمكن توسيعه ليشمل:

- **نماذج حكومية:** طلبات دعم، تسجيل منشآت، استمارات صحية وتعليمية
- **القطاع المالي:** طلبات تمويل، نماذج فتح حسابات، مطالبات تأمين
- **أي استمارة إلكترونية** فيها حقول مترابطة منطقيًا

**آلية التوسع:**
- إضافة قوالب قواعد منطقية جديدة لكل نوع استمارة
- دعم لغات متعددة (العربية/الإنجليزية) من خلال قدرات الـ LLM
- النشر على بنية سحابية لخدمة آلاف الطلبات بالتوازي
- ترقية النموذج اللغوي المستخدم دون التأثير على بقية النظام

## 9. الأثر على المستفيدين

| المستفيد | الأثر |
|----------|-------|
| **الباحثون الميدانيون** | تنبيهات لحظية أثناء التعبئة — تقلّل الحاجة لإعادة الزيارات |
| **محللو البيانات** | بيانات أنظف من المصدر — وقت أقل في التنظيف اليدوي |
| **أقسام الإحصاء** | مؤشرات أدق وأسرع في الإصدار |
| **صنّاع القرار** | قرارات مبنية على بيانات موثوقة (توجيه دعم، توزيع موارد) |
| **المستجيبون** | تجربة أفضل — النظام يساعدهم على تصحيح الأخطاء فورًا |

---

## 10. الأثر المالي والتشغيلي

- **خفض تكلفة التنظيف اليدوي:** اكتشاف الأخطاء لحظة الإدخال يُلغي ساعات عمل التدقيق اللاحق
- **تقليل إعادة الزيارات الميدانية:** كل زيارة مُعادة = تكلفة نقل + وقت باحث + إزعاج للمستجيب
- **تسريع إصدار المؤشرات:** بيانات نظيفة من المصدر = وقت أقل بين الجمع والنشر
- **تقليل هدر الإنفاق:** قرارات مبنية على بيانات دقيقة = توزيع موارد أكفأ
- **إمكانية التحول لخدمة مدفوعة (SaaS):** تقديم الحل للجهات الأخرى كخدمة اشتراك

---

## 11. الأثر على الهيئة

- يعزز **دقة بيانات الهيئة** وجودة مؤشراتها الإحصائية
- يقلل **تكلفة ضبط الجودة** بأتمتة التحقق المنطقي
- يُرسّخ صورة الهيئة كجهة **رائدة في توظيف الذكاء الاصطناعي** لتحسين العمليات
- تخزين سجلات الأخطاء يسمح بـ **تعلم مستمر** — كل خطأ يُكتشف يُحسّن القواعد مستقبلًا
- بنية مرنة تسمح بترقية النموذج اللغوي أو توسيع التغطية **بدون إعادة بناء النظام**
